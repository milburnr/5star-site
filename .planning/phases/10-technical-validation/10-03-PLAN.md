<task type="auto">
  <name>Task 1: Run internal link check and fix any broken links</name>
  <files></files>
  <action>
    1. Run the internal link checker created in Plan 10-01:
       ```
       node scripts/audit-internal-links.js
       ```

    2. If PASS (exit 0): Document the clean result and proceed to Task 2.

    3. If FAIL (exit 1): For each broken internal link reported:
       a. Identify the source page(s) in app/ that contain the broken href
       b. Determine the correct target:
          - If the target page was deleted in Phase 2 (URL canonicalization), update the href to the canonical URL
          - If the target page was renamed or moved, update to new path
          - If the link points to a page that never existed, remove the link or point to the nearest valid parent (e.g., /services/)
       c. Fix the href in the source page.tsx file(s)

    4. After fixing, rebuild and re-run:
       ```
       npm run build
       node scripts/audit-internal-links.js
       ```
       Repeat until exit code 0.

    5. Common broken link patterns from prior phases:
       - Links to `-texas-` or `-tx-` variant URLs deleted in Phase 2 (should point to canonical)
       - Links to hub pages that didn't exist before Phase 6 (these should now resolve)
       - Breadcrumb links to non-existent parent paths
       - Links in InternalLinks component to pages that were redirected

    Note: If broken links are found AND fixed, also re-run `node scripts/validate-schema.js` and `node scripts/audit-meta.js` to confirm no regressions.
  </action>
  <verify>
    ```
    node scripts/audit-internal-links.js
    ```
    Exit code 0: "PASS: All internal links resolve"
  </verify>
  <done>Internal link checker passes with zero broken links. If fixes were needed, build is clean and other validation scripts still pass.</done>
</task>

<task type="auto">
  <name>Task 2: Resubmit all pages to Google Indexing API</name>
  <files></files>
  <action>
    1. First, run a dry-run to verify the URL list is correct:
       ```
       node submit-to-indexing-api.js --dry-run
       ```
       Verify the URL count matches the sitemap count (~216). Review the URL list for any obviously wrong entries.

    2. Check that the Google service account key file exists:
       ```
       ls -la google-service-account-key.json
       ```

    3. If the key file exists, submit all pages:
       ```
       node submit-to-indexing-api.js
       ```
       The script has built-in rate limiting (100ms delay, batch size 10). With ~216 pages, this should complete in ~25 seconds and stay within the 200 URLs/day limit.

    4. If submission hits rate limits (429 errors):
       - Document how many URLs were successfully submitted
       - Note remaining URLs for submission the following day
       - This is NOT a blocker — document and move on

    5. If the key file does NOT exist or auth fails:
       - Document the error clearly
       - Record this as a manual follow-up item: "User needs to place google-service-account-key.json and re-run node submit-to-indexing-api.js"
       - This is NOT a blocker per phase constraints ("Google Indexing API submission is optional if credentials/quota issues arise")

    6. Document the final submission results:
       - Total pages submitted successfully
       - Any failures and reasons
       - Any pages deferred to next day due to quota
  </action>
  <verify>
    ```
    node submit-to-indexing-api.js --dry-run
    ```
    Dry-run shows all 216+ URLs. If credentials available, actual submission reports success count matching total page count (or close to it with quota limits).
  </verify>
  <done>Google Indexing API dry-run confirms correct URL list. Actual submission either completes successfully or is documented as blocked by credentials/quota (non-blocking per phase constraints).</done>
